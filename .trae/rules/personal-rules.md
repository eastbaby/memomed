# 个人编码规则

## 1. 代码风格

### 1.1 TypeScript/JavaScript
- 使用 2 空格缩进
- 使用单引号 `'` 而不是双引号 `"`
- 每行代码长度不超过 80 个字符
- 使用分号 `;` 结尾
- 大括号 `{` 与语句在同一行
- 箭头函数优先于普通函数
- 使用 `const` 和 `let`，避免使用 `var`

### 1.2 Python
- 使用 4 空格缩进
- 每行代码长度不超过 79 个字符
- 使用蛇形命名法 `snake_case`
- 类名使用驼峰命名法 `CamelCase`
- 导入语句按标准库、第三方库、本地库的顺序排列

## 2. 命名约定

### 2.1 变量和函数
- 变量：使用描述性名称，避免单个字母（循环变量除外）
- 函数：使用动词开头，如 `getUser()`、`calculateTotal()`
- 常量：使用全大写，如 `MAX_SIZE`、`API_BASE_URL`

### 2.2 组件和文件
- 组件名：使用 PascalCase，如 `UserProfile.tsx`
- 文件：使用小写加连字符，如 `user-service.ts`
- 目录：使用小写加下划线，如 `src/components/`

## 3. 开发流程

### 3.1 版本控制
- 每次提交只包含一个功能或修复
- 提交信息使用语义化格式：`type(scope): subject`
- 常用 type：feat（新功能）、fix（修复）、docs（文档）、style（样式）、refactor（重构）、test（测试）
- 定期拉取最新代码，避免冲突

### 3.2 测试
- 新功能必须添加相应的测试用例
- 测试覆盖率目标：核心功能 ≥ 80%
- 运行完整测试套件后再提交代码

### 3.3 代码审查
- 提交前自查代码质量
- 遵循团队代码审查流程
- 及时回应代码审查反馈

## 4. 性能优化

### 4.1 前端优化
- 组件懒加载：`React.lazy()`
- 图片优化：使用适当尺寸和格式
- 减少不必要的重渲染
- 使用 `useMemo` 和 `useCallback` 优化计算和函数

### 4.2 后端优化
- 数据库查询优化：使用索引，避免全表扫描
- 缓存策略：合理使用缓存减少重复计算
- 异步处理：使用 `async/await` 和非阻塞操作
- 代码分割：将大型函数拆分为更小的函数

## 5. 安全性

### 5.1 前端安全
- 避免 XSS 攻击：使用 React 的自动转义
- 避免 CSRF 攻击：使用 CSRF token
- 敏感信息不硬编码在前端

### 5.2 后端安全
- 输入验证：验证所有用户输入
- 密码存储：使用 bcrypt 等安全哈希算法
- API 认证：使用 JWT 或 OAuth
- 权限控制：实施最小权限原则

## 6. 文档

### 6.1 代码注释
- 复杂函数必须添加 JSDoc 或 docstring
- 关键逻辑添加行注释
- 避免冗余注释，代码本身应该自解释

### 6.2 项目文档
- README.md：项目概述、安装和运行说明
- API 文档：使用 Swagger 或类似工具
- 架构文档：系统设计和组件关系

## 7. 工具和配置

### 7.1 编辑器配置
- 使用 VS Code 作为主要编辑器
- 安装必要的扩展：ESLint、Prettier、GitLens
- 配置 `.vscode/settings.json` 保持一致的编辑体验

### 7.2 构建工具
- 前端：Vite + TypeScript
- 后端：FastAPI + Pydantic
- 包管理：npm（前端）、uv（后端）

## 8. 最佳实践

### 8.1 代码组织
- 按功能模块组织代码
- 遵循单一职责原则
- 保持文件大小合理（单个文件不超过 500 行）

### 8.2 错误处理
- 使用 try-catch 捕获异常
- 提供有意义的错误信息
- 日志记录：记录关键操作和错误

### 8.3 可维护性
- 避免硬编码值
- 使用配置文件管理环境变量
- 编写可测试的代码
- 定期重构代码以提高质量

## 9. 学习和提升

### 9.1 持续学习
- 每周学习新的技术和工具
- 阅读技术博客和文档
- 参与社区讨论和开源项目

### 9.2 代码质量
- 使用 ESLint 和 Prettier 保持代码风格一致
- 定期进行代码审查和重构
- 学习设计模式和最佳实践

## 10. 团队协作

### 10.1 沟通
- 清晰表达技术需求和问题
- 及时更新任务状态
- 尊重团队成员的意见

### 10.2 协作工具
- 使用 Git 进行版本控制
- 使用 Issue 跟踪任务和 bug
- 使用 Pull Request 进行代码审查

这些规则旨在帮助我保持代码质量和开发效率，同时确保项目的可维护性和安全性。我会根据实际项目需求和技术发展不断更新这些规则。
